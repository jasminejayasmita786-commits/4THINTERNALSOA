Django's MTV Architecture

What is MTV?

Django follows the MTV (Model-Template-View) pattern — a variation of MVC where the Template acts as the View and Django itself acts as the Controller.

  • Model — Defines data structure. Handles database interaction.
  • Template — Handles presentation (HTML + Django Template Language).
  • View — Contains business logic. Fetches data from Model, sends to Template.
✏️ Write This In Exam

Django follows the MTV pattern: Model (data layer), Template (presentation layer), View (business logic layer). When a browser sends an HTTP request, Django's URL Dispatcher routes it to the correct View, which queries the Model and renders a Template to return an HTTP response.

What is ORM?

ORM (Object Relational Mapping) allows us to interact with the database using Python code without writing SQL queries. Django ORM maps Python classes → Database tables, class attributes → columns, and objects → rows.

from django.db import models
from django.utils import timezone
from django.conf import settings

class Post(models.Model):

    class Status(models.TextChoices):
        DRAFT = 'DF', 'Draft'
        PUBLISHED = 'PB', 'Published'

    title = models.CharField(max_length=250)
    slug = models.SlugField(max_length=250, unique_for_date='publish')
    author = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name='blog_posts'
    )
    body = models.TextField()
    publish = models.DateTimeField(default=timezone.now)
    created = models.DateTimeField(auto_now_add=True)
    updated = models.DateTimeField(auto_now=True)
    status = models.CharField(
        max_length=2,
        choices=Status,
        default=Status.DRAFT
    )

    class Meta:
        ordering = ['-publish']
        indexes = [models.Index(fields=['-publish'])]

    def __str__(self):
        return self.title

    def get_absolute_url(self):
        from django.urls import reverse
        return reverse(
            'blog:post_detail',
            args=[self.publish.year, self.publish.month, self.publish.day, self.slug]
        )

Pagination

What is Pagination?

Pagination means splitting a large list of records into multiple pages. Django has a built-in Paginator class. Example: 10 posts → 3 per page → 4 pages total.

blog/views.py — Pagination View PYTHON 
from django.core.paginator import Paginator, EmptyPage, PageNotAnInteger
from django.shortcuts import render
from .models import Post

def post_list(request):
    post_list = Post.published.all()
    paginator = Paginator(post_list, 3)          # 3 posts per page
    page_number = request.GET.get('page', 1)    # get ?page= from URL
    try:
        posts = paginator.page(page_number)
    except PageNotAnInteger:
        posts = paginator.page(1)               # non-integer → go to page 1
    except EmptyPage:
        posts = paginator.page(paginator.num_pages)  # out of range → last page
    return render(request, 'blog/post/list.html', {'posts': posts})
templates/pagination.html HTML 
<div class="pagination">
  <span class="step-links">
    {% if page.has_previous %}
      <a href="?page={{ page.previous_page_number }}">Previous</a>

    </span>

At the bottom of list.html, inside the content block:
{% include "pagination.html" with page=posts %}

Class-Based Views (ListView)

Why Class-Based Views?

Class-Based Views (CBV) provide a structured, reusable way to write views. Django provides generic views like ListView, DetailView, etc. that handle common tasks automatically.

blog/views.py — Class-Based ListView PYTHON 
from django.views.generic import ListView
from .models import Post

class PostListView(ListView):
    queryset = Post.published.all()          # data to display
    context_object_name = 'posts'          # variable name in template
    paginate_by = 3                         # auto pagination
    template_name = 'blog/post/list.html'  # template to use
blog/urls.py — CBV URL PYTHON 
path('', views.PostListView.as_view(), name='post_list'),
# .as_view() converts class into callable view function
⚠️ When using CBV with pagination, use page=page_obj in the include tag (not page=posts). This is a common exam mistake!

blog/views.py — post_share view PYTHON 
from django.core.mail import send_mail
from .forms import EmailPostForm

def post_share(request, post_id):
    post = get_object_or_404(Post, id=post_id, status=Post.Status.PUBLISHED)
    sent = False

    if request.method == 'POST':
        form = EmailPostForm(request.POST)
        if form.is_valid():
            cd = form.cleaned_data
            post_url = request.build_absolute_uri(post.get_absolute_url())
            subject = f"{cd['name']} recommends you read {post.title}"
            message = f"Read {post.title} at {post_url}\n\n{cd['name']}'s comments: {cd['comments']}"
            send_mail(subject=subject, message=message,
                      from_email=None, recipient_list=[cd['to']])
            sent = True
    else:
        form = EmailPostForm()

    return render(request, 'blog/post/share.html', {'post': post, 'form': form, 'sent': sent})

    post_list = Post.published.all()
    tag = None
    if tag_slug:
        tag = get_object_or_404(Tag, slug=tag_slug)
        post_list = post_list.filter(tags__in=[tag])
    # ... pagination ...

def post_detail(request, year, month, day, post):
    # ... get post ...
    # Similar posts based on tags
    post_tags_ids = post.tags.values_list('id', flat=True)
    similar_posts = Post.published.filter(
        tags__in=post_tags_ids
    ).exclude(id=post.id)
    similar_posts = similar_posts.annotate(
        same_tags=Count('tags')

1 The method used in Django models to return the canonical URL of an object is ________. CO1L1
Answer: get_absolute_url()

The get_absolute_url() method is defined inside a Django model. It uses reverse() to dynamically generate the canonical URL of a model object. It is commonly used in templates as {{ post.get_absolute_url }}.
2 Which Django function is used to dynamically generate URLs using URL patterns? CO1L2
  • a) render()
  • b) reverse()
  • c) include()
  • d) redirect()
Answer: b) reverse()

reverse('blog:post_detail', args=[year, month, day, slug]) generates a URL dynamically from URL name patterns. It avoids hardcoding URLs in code.

5 Explain the purpose of canonical URLs in Django applications. CO2L3
Answer (write in 3–4 lines for exam):

A canonical URL is the definitive or main URL for a specific object/page. In Django, it is implemented using the get_absolute_url() method in the model, which uses reverse() to generate the URL dynamically. It ensures that links in templates always point to the correct page, prevents duplicate URLs, and improves SEO by establishing the authoritative URL for each piece of content.
6 What does the following URL pattern represent? path('<int:year>/<int:month>/<int:day>/<slug:post>/', views.post_detail) CO2L3
  • a) Static URL
  • b) SEO-friendly dynamic URL
  • c) Admin URL
  • d) API endpoint
Answer: b) SEO-friendly dynamic URL

This URL contains the publish year, month, day, and slug. Example: /2026/04/17/my-first-post/. It is called SEO-friendly because search engines can understand the content from the URL structure itself. The int: and slug: are URL converters that validate the data type.
7 After installing django-taggit, which setting file is modified to register the application? CO3L1
Answer: settings.py

After installing with pip install django-taggit, add 'taggit' to the INSTALLED_APPS list in settings.py. Then run python manage.py migrate to create taggit's database tables.
8 What does fail_silently=False do in send_mail()? CO3L2
  • A) Hides errors
  • B) Raises errors if email fails
  • C) Sends silently
  • D) Ignores email
Answer: B) Raises errors if email fails

fail_silently=False means Django will raise an exception if the email sending fails (e.g., SMTP error). If set to True, errors are suppressed silently. Setting it to False is useful during development to debug email issues.
9 Write the Django ORM query to retrieve similar posts based on common tags using Count(). CO3L4
Answer:
views.py
from django.db.models import Count

# Get IDs of tags of current post
post_tags_ids = post.tags.values_list('id', flat=True)

# Get posts that share tags, exclude current post
similar_posts = Post.published.filter(
    tags__in=post_tags_ids
).exclude(id=post.id)

# Annotate with count of shared tags, order by it
similar_posts = similar_posts.annotate(
    same_tags=Count('tags')
).order_by('-same_tags', '-publish')[:4]

Part (a) — Pagination View

from django.core.paginator import Paginator, EmptyPage, PageNotAnInteger
from django.shortcuts import render
from .models import Post

def post_list(request):
    post_list = Post.published.all()
    paginator = Paginator(post_list, 3)
    page_number = request.GET.get('page', 1)
    try:
        posts = paginator.page(page_number)
    except PageNotAnInteger:
        posts = paginator.page(1)
    except EmptyPage:
        posts = paginator.page(paginator.num_pages)
    return render(request, 'blog/post/list.html', {'posts': posts})

Function-Based View

views.py
from django.shortcuts import render, get_object_or_404
from .models import Post

def post_list(request):
    posts = Post.published.all()
    return render(request,
        'blog/post/list.html',
        {'posts': posts})

def post_detail(request, year, month, day, post):
    post = get_object_or_404(Post,
        status=Post.Status.PUBLISHED,
        slug=post,
        publish__year=year,
        publish__month=month,
        publish__day=day)
    return render(request,
        'blog/post/detail.html',
        {'post': post})

Common Exam Mistakes to Avoid
❌ Using page=posts instead of page=page_obj in ListView pagination include tag.
❌ Forgetting to run makemigrations AND migrate after changing models.
❌ Using auto_now=True for created field (it's auto_now_add=True) — auto_now updates on every save!
❌ Forgetting app_name = 'blog' in urls.py when using namespaced URLs like blog:post_detail.
❌ Using form.save() directly for comments — always use commit=False to set the post first.
MTV Request Flow (Draw in Exam)
✏️ Exam Answer — Django Request-Response Flow

1. Browser sends HTTP Request to Django server.
2. Django's URL Dispatcher matches the URL pattern in urls.py.
3. The matched View function/class is executed.
4. View queries the Model (via ORM) to get data from database.
5. Model returns data to the View.
6. View renders the Template with the data.
7. Template generates HTML and View returns HTTP Response to browser.
